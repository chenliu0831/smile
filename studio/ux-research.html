Turning a flat report viewer into an explorable AutoML cockpit

Five consolidated feature & UX improvements to enrich the pipeline, leaderboard, and EDA experience — give the user interactive access to steer and interrogate the run. Grounded in a real completed run (smile_example, Titanic), and stress-tested by an adversarial feasibility pass that caught hallucinated numbers, a wrong feature-join, and architecture violations.

The core gap — it's a transport problem, not a rendering one

Clair's run writes a goldmine of structured data — per-model AUCs and tuned hyperparameters, calibration and permutation importance with error bars, and 179 rows of per-row prediction probabilities. But RunArtifactWatcher.java is deliberately file-existence-only and surfaces just flat markdown + base64 PNGs. So the UI shows a static timeline, a read-only 5-row leaderboard hard-coded to "binary", and a dead path string where the richest predictive artifact should be.

The unlock: three of those files sit in output/ as the skill's intentional public outputs, so they can be surfaced without touching the private state.json the ADR-0005/0006 boundary protects. Surface the public JSON/CSV first, then build interactive consumers on top, and reuse the agentic user-message channel that already exists rather than inventing new commands.

final_metrics.json — OOF/test AUC, per-model, ensemble vs stacking postprocess_results.json — calibration + perm-importance ±std best_params.json — tuned hyperparameters per model final/submission.csv — 179× proba / pred / actual summary.md — the merged 10-model ranking automl_report.md — 11 steps, refinement Δ, next steps

The Top 5 — consolidated initiatives

Each bundles a surfacing enabler with its highest-value consumer(s). Ordered by leverage. Honesty notes come straight from the adversarial verifiers.

1

Predictions Studio — ROC, confusion matrix & a live threshold slider

diagnostics ★ flagship · 9.5 effort M impact high

The signature interactive view. final/submission.csv reaches the UI today as a dead 📄 path string — yet every per-row probability + label is a ROC / confusion / threshold / error-inspection goldmine, all computable client-side once the rows cross the wire. This is the single highest-leverage enabler in the whole set.

Bundles
Predictions dataframe (enabler) Threshold Studio Probability-separation histogram Misclassified-passenger inspector

What you build

A two-pane panel: a 2×2 confusion matrix beside a ROC curve, both recomputed in the browser as the user drags one threshold slider. Live readouts of accuracy / precision / recall / F1. Click a confusion cell → list those rows. Click a misclassified passenger → expand its features.

Data that powers it

submission.csv: Survived_proba, Survived_pred, Survived_actual (179 rows). Anchored to final_metrics.json test_auc / f1 and state.json test_accuracy so the client-computed curve is validated against the daemon's own numbers.

Honesty note (from the verifier) Threshold tuning on the hold-out is in-sample — frame "Maximize F1" as a what-if, not validated, and open the slider at the model's real ~0.50 operating point (not the 0.42 an early draft hallucinated). 2

Live Run Scorecard — a persistent metric strip that self-configures the UI

pipeline · surfacing ★ flagship · 9.2 effort M impact high

Every numeric score the agent computes dies in JSON. A persistent scorecard strip above the canvas turns final_metrics.json into a live header — and the task_type it carries finally kills the hard-coded "binary" in the leaderboard, letting the whole UI self-configure for regression / multiclass runs.

Bundles
Metrics artifact (enabler) Run context header task_type → defaultMetric()

What you build

A sticky strip: problem type · primary metric · CV · rows, then the headline scores that fill in as the run completes. Hover a metric → its source field + which step produced it. The task_type feeds defaultMetric() so metric labels are correct for any task, not just binary.

Data that powers it

final_metrics.json: oof_auc 0.895, test_auc 0.860, test_acc 0.827, test_f1 0.763, method weighted_average. task_type + primary_metric parsed from automl_report.md problem-setup.

┌ binary_classification · AUC-ROC · 5-fold · 891 rows ──────────── ✓ complete ┐ │ OOF 0.895 TEST 0.860 ACC 0.827 F1 0.763 Brier 0.137 │ │ ensemble: weighted_avg (xgb·lgbm·rf) · vs stacking 0.892 │ └────────────────────────────────────────────────────────────────────────────┘
Honesty note Source the scorecard from public final_metrics.json, not the private state.json — reaching into the skill's checkpoint reverses a documented ADR-0005/0006 decision and couples the wire contract to an internal schema that can drift. Carry the parsed JSON in the existing nullable body string + one new "metrics" ArtifactKind literal to avoid churning the positional Java Artifact record.
Wiring   Contract: add "metrics" to ArtifactKind (TypeBox), regenerate. Watcher: read final_metrics.json on its existing tick, emit a metrics artifact (JSON in body). New Scorecard.tsx; defaultMetric() consumes task_type.
3

Interactive Leaderboard — sortable, score bars, per-model drill-down

leaderboard 8.8 effort M impact high

Today Leaderboard.tsx parses only the 5 default candidates, has no sort, no bars, and never shows the 3 tuned models or 2 ensembles. The cheapest unlock: repoint it at summary.md — which already is the merged 10-row ranking — then make it interactive and drillable.

Bundles
Unified sortable board (summary.md) Per-model hyperparameter card Ensemble-vs-stacking verdict Runtime-vs-accuracy scatter

What you build

Sortable columns, horizontal score bars (gold = ensemble, blue = tuned, grey = default), and conditional ±std whiskers. Expand a row → tuned hyperparameters from best_params.json with a "copy as Python dict" and a default-vs-tuned delta. A grouped block shows the weighted-avg ensemble beat every base learner and stacking.

Data that powers it

summary.md 10-row table (OOF AUC, std, notes); best_params.json per-model params + score; final_metrics.json ensemble weighted_avg 0.8946 vs stacking 0.8924.

Honesty note Tuned/ensemble rows have std = "—" — render the whisker only where std is finite. The files disagree on per-model AUC (tuning-CV 0.8955 vs OOF 0.8913 vs default 0.8918): pin every number to one source and compute lift client-side so the arithmetic always reconciles. Emit best_params.json as its own companion artifact keyed by the agent's xgb/lgbm/rf names to sidestep the xgb≠xgboost join bug; badge logreg/mlp "tuned only".
Wiring   v1 is zero contract/Java change: surface summary.md as kind:"leaderboard", teach the parser its column layout, add a useState sort key + bars in Leaderboard.tsx. Drill-down adds one watcher read of best_params.json as a companion artifact.
4

Driver Diagnostics — permutation importance with uncertainty whiskers

eda · diagnostics 9.0 · 8.6 effort S impact high

The real diagnostic payoff on this dataset. postprocess_results.json carries each driver's importance mean AND std — the whole point of permutation importance (is the ranking stable?) — but it renders today as a flat numbered list with no magnitudes. A bar chart with error whiskers makes stability legible at a glance, and each bar becomes an EDA entry point.

Bundles
Diagnostics artifact (enabler) Importance bars + whiskers "Ask Clair about this feature" "Slice survival by feature"

What you build

A horizontal bar chart (mean ΔAUC, sorted), each bar with a ±1 std whisker, in plain SVG — no ECharts, no Arrow. Hover → exact mean±std + plain-English line. Click a feature → templated "Ask Clair about Title_Mr" turn, or "Slice survival by it" runs a grouped SELECT over the feature table inline.

Data that powers it

postprocess_results.json top5_features: Title_Mr 0.066±0.018, Sex_female 0.047±0.016, Age 0.038±0.014, Title_Mrs 0.025±0.005 (stable), FarePerPerson 0.018±0.010 (noisy).

Honesty note Whiskers aren't in any existing chart path — render them in plain SVG, don't try to reuse the DataVizSpec bar (no whisker channel, by-reference data). Carry the 5-row array inline in the new diagnostics artifact body (tiny — no DuckDB round-trip). Defer the "remove feature & re-run" button until agent-side feature exclusion is confirmed.

5

Close the Loop — passive readout → active steering

interactivity · eda 8.3 · 8.0 effort S impact high

The agentic channel (user-message) already exists, but every view is a dead end. Two cheap, high-leverage steering wins reuse that exact path — making the EDA and the report actionable instead of read-only.

Bundles
Ask Clair about this column One-click Next Steps

Ask Clair about this column

An affordance on every SqlConsole schema-rail column, pre-seeding a prompt with the column's name, dtype, and — when present — its driver rank from postprocess_results.json. "How does Age relate to survival?" → Clair answers in chat and proposes a confirming SQL into the editor. Keep insert-then-Run (don't fight the agent for the single DuckDB connection).

One-click Next Steps

Parse the report's own Recommended Next Steps into context-rich buttons — "Add CatBoost as a 4th learner", "Switch Platt → isotonic", "Tune threshold for F1" — each sending a templated sendMessage carrying the current solution + baseline AUC (0.8946) as a new turn.

[histogram of Age] NEXT STEPS (from Clair's report) [ ? Ask Clair about Age ] [+ Add CatBoost as 4th learner] → "Age, driver #3, [Switch Platt → isotonic calibration] perm-imp 0.038 …" [K-fold target-mean encode Title] Clair: "children survived [Tune threshold for F1] at ~2× …" + SQL into editor ─ each runs as a new steering turn
Honesty note Scope "Ask Clair" to the schema rail that exists — flat-PNG charts can't carry per-bar click targets, so "ask about this histogram bar" is a later dependency, not this ticket. Ship Next Steps as steering shortcuts, not an auto-landing-Δ-row promise: the one load-bearing backend follow-up is giving the report path the same mtime re-emit key the image path already has, so a regenerated report refreshes the canvas.
Wiring   Almost entirely frontend — reuses askClair / sendMessage / awaitingAgentSql. Driver context needs postprocess_results.json surfaced (shared with #4). One backend line: mtime re-emit for the report artifact in RunArtifactWatcher.
